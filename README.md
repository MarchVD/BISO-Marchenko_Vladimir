### Описание предметной области
#### Назначение системы
Интернет-магазин предназначен для продажи различных товаров. Система обеспечивает взаимодействие между продавцами и покупателями, а также позволяет администраторам контролировать работу магазина.

#### Список акторов с кратким описанием их роли
**Покупатель** - Пользователь, который просматривает каталог товаров, оформляет и оплачивает заказы
**Продавец** - Пользователь, который публикует товары, собирает и отправляет заказы покупателям
**Администратор** - Управляет учетными записями пользователей и контролирует работу системы

#### Список вариантов использования
**Регистрация** - создание новой учетной записи пользователем
**Вход в систему** - авторизация пользователя в системе
**Просмотр каталога** - просмотр списка доступных товаров
**Публикация товара** - добавление нового товара продавцом
**Составление заказа** - формирование заказа из выбранных товаров
**Оплата заказа** - проведение оплаты заказа через платежную систему
**Сбор заказа** - комплектация заказа продавцом
**Управление пользователями** - управление учетными записями администратором

#### Ключевой сценарий для диаграммы последовательности
**Сценарий:** Оформление и оплата заказа покупателем
1. Покупатель добавляет товары в корзину и нажимает "Оформить заказ"
2. Система проверяет наличие всех товаров
3. Если товары доступны, система создает заказ в базе данных
4. Покупатель переходит к оплате заказа
5. Система обрабатывает оплату через платёжную систему
6. После успешной оплаты статус заказа обновляется
7. Покупатель получает подтверждение оформления заказа

### Ответы на контрольные вопросы
#### 1. Какие типы диаграмм UML относятся к структурным, а какие к поведенческим?
**Структурные диаграммы** - Class Diagram, Component Diagram, Deployment Diagram
**Поведенческие диаграммы** - Use-Case Diagram, Sequence Diagram

#### 2. В чём разница между отношениями include и extend?
**include** - обязательное выполнение включаемого варианта
**extend** - опциональное выполнение расширяющего варианта

#### 3. Чем агрегация отличается от композиции?
**Пример из моей системы:**
- **Композиция:** `Order *-- OrderItem` — позиции заказа не могут существовать без самого заказа (сильная связь)
- **Агрегация:** `Product o-- OrderItem` — товар может существовать без позиции заказа (слабая связь)

#### 4. Что такое кратность (multiplicity) в диаграмме классов?
**Кратность** показывает, сколько объектов одного класса может быть связано с одним объектом другого класса.
```plantuml
Seller "1" -- "many" Product : publishes
Buyer "1" -- "many" Order : makes
Order "1" -- "0..1" Delivery : delivered
```
Значения:
- `1` — ровно один
- `0..1` — ноль или один
- `many` — любое количество
- `1..*` — один и более

#### 5. Как в диаграмме классов отобразить структуру (struct) в Go? Что такое стереотип struct?
Стереотип `<<struct>>` показывает, что это не класс в классическом ООП-смысле, а структура данных языка Go.
```plantuml
class "OrderItem" as OrderItem <<struct>> {
    -item: Product
    -quantity: int
    -price: float
}
```

#### 6. Чем встраивание (embedding) в Go отличается от наследования в классических ООП-языках?
- Наследование - отношение "является" (is-a)
- Встраивание - отношение "имеет" (has-a)
```plantuml
' Наследование
User <|-- Buyer
' Встраивание
User <|-- Buyer : встраивание
```

#### 7. Какие типы сообщений используются в диаграмме последовательности и в чём их отличие?
**Синхронное** `->` - вызов с ожиданием ответа
**Асинхронное** `->>` - вызов без ожидания ответа
**Возвратное** `-->` - ответ на синхронный вызов

#### 8. Для чего используются комбинированные фрагменты (alt, loop, opt) в диаграмме последовательности?
`alt` - ветвление (если-иначе), `loop` - цикл, `opt` - опциональный блок

#### 9. В чём принципиальное отличие диаграммы компонентов от диаграммы классов?
Диаграмма классов отражает логическую структуру, диаграмма компонентов отражает физическую структуру

#### 10. Какие элементы присутствуют на диаграмме развёртывания? Что такое артефакт в контексте Go-приложения?
На диаграмме развёртывания присутствует:
- **Узел (node)** — физический или виртуальный сервер
- **Артефакт (artifact)** — физический файл или компонент
- **Коммуникационный путь** — канал связи между узлами
**Артефакт в Go-приложении:** - скомпилированный бинарный файл (`api-server`, `business-service`), конфигурационные файлы

#### 11. Что такое подход «Diagram as Code» и каковы его преимущества перед графическими редакторами?
**Diagram as Code** — подход, при котором диаграммы описываются в текстовом виде и после генерируются в изображения. Важными преймуществами является лёгкая модификация и контроль изменений через Git

#### 12. Как в VS Code открыть предпросмотр диаграммы PlantUML?
`Alt + D` (Windows/Linux) или `Option + D` (Mac)

#### 13. В каких форматах можно экспортировать диаграммы из VS Code с помощью расширения PlantUML?
Можно экспортировать в форматы PNG, SVG, PDF и другие
